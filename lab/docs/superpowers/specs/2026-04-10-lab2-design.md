# Lab2 设计规格

日期：2026-04-10

## 1. 目标

在 `lab/lab2` 下新建一个完整独立实验模块，基于 `lab1` 的 MPI 点对点矩阵乘法思路，完成 lab2 题目要求，并额外交付数据分析与可视化支撑材料。最终产物需要形成以下闭环：

- MPI 代码实现
- benchmark 数据生成
- 自动绘图
- 报告草稿与分析框架

本次明确目标：

1. 实现两种 MPI 版本：点对点对照版与集合通信进阶版
2. 使用 `MPI_Type_create_struct` 聚合运行配置并通信
3. 批量生成 `1/2/4/8/16` 进程、`128/256/512/1024/2048` 规模下的性能数据
4. 生成适合课程报告的可视化图表
5. 提供可直接承载内容的报告草稿
6. 代码结构预留第三版扩展空间，但本次不实现第三版

## 4. 实现版本设计

### 4.1 `mpi_v1_p2p`

该版本作为对照组，用于承接 `lab1` 的点对点通信思路，并与集合通信版本进行横向对比。

核心流程：

1. rank 0 生成完整矩阵 `A` 和 `B`
2. 按行切分 `A`
3. 使用 `MPI_Send/MPI_Recv` 分发 `A` 的子块
4. 使用 `MPI_Send/MPI_Recv` 发送完整 `B`
5. 各 rank 执行本地矩阵乘法
6. 使用 `MPI_Send/MPI_Recv` 回传 `C` 子块
7. rank 0 汇总结果并输出结构化指标

输出版本名：

- `version=mpi_v1_p2p`

### 4.2 `mpi_v2_collective`

该版本是 lab2 主实现，要求使用 MPI 集合通信完成主要进程间通信，并使用 `MPI_Type_create_struct` 聚合进程内变量后通信。

核心流程：

1. rank 0 构造运行配置结构体，包含至少：
   - `m`
   - `n`
   - `k`
   - `seed`
   - `dump_matrix`
2. 使用 `MPI_Type_create_struct` 为该配置结构体建立 MPI datatype
3. 使用该 datatype 从 rank 0 向所有进程广播运行配置
4. 各 rank 根据配置独立计算 `row_counts` 与 `row_displs`
5. rank 0 生成完整 `A`、`B`
6. 使用 `MPI_Scatterv` 分发 `A` 的行块
7. 使用 `MPI_Bcast` 广播完整 `B`
8. 各 rank 执行本地矩阵乘法
9. 使用 `MPI_Gatherv` 汇总 `C`
10. rank 0 输出结果与必要统计信息

输出版本名：

- `version=mpi_v2_collective`

### 4.3 第三版扩展预留

本次不实现第三版，但在 `matmul_mpi.hpp` 中预留以下扩展边界：

- 运行配置封装
- 行划分函数
- MPI datatype 构建函数
- 统一输出函数
- 本地计算核

这样后续若需要新增如 `v3_collective_transposed_b` 或其他优化版，不需要重构目录结构。

## 5. 公共代码设计

`src/matmul_mpi.hpp` 将承接和演化 `lab1/src/matmul_mpi.hpp` 的公共逻辑，包括：

- 参数解析
- 随机矩阵生成
- checksum 计算
- `max_abs` 计算
- 小规模矩阵打印
- 本地串行乘法核
- 行切分辅助函数
- `MPI_Type_create_struct` 构建辅助函数
- 统一结果输出

设计原则：

- 保持单一职责
- 对两个版本共享的逻辑沉淀到公共头文件
- 不为尚未实现的第三版引入过度抽象

## 6. 输出契约

所有 MPI 可执行程序继续沿用 `lab0` 和 `lab1` 的稳定 `key=value` 风格。至少输出：

- `version`
- `m`
- `n`
- `k`
- `seed`
- `processes`
- `time_sec`
- `checksum`
- `max_abs`

可额外添加但不得破坏解析稳定性：

- `impl`
- `comm_mode`
- `partition`

`benchmark.py` 必须以这套契约为解析基础。

## 7. Benchmark 设计

`lab/lab2/scripts/benchmark.py` 负责统一跑两版程序，并生成课程需要的数据与绘图友好格式。

固定实验维度：

- 版本：`mpi_v1_p2p`、`mpi_v2_collective`
- 进程数：`1, 2, 4, 8, 16`
- 矩阵规模：`128, 256, 512, 1024, 2048`
- 重复次数：默认 `3`
- 默认 seed：`20250401`

输出文件：

- `results/benchmark_results.json`
- `results/performance_table.csv`
- `results/summary_by_version.csv`

其中：

- `benchmark_results.json` 保存每次运行的原始记录、stdout/stderr、解析结果与错误状态
- `performance_table.csv` 用于课程表格填写
- `summary_by_version.csv` 供绘图与报告使用

### 7.1 健壮性要求

benchmark 需要显式区分：

- 程序运行失败
- 输出字段缺失
- 同配置重复运行 checksum 不一致
- 与 1 进程基线 checksum 不一致

结果状态至少包括：

- `ok`
- `failed`
- `partial_failure`
- `checksum_mismatch`
- `baseline_checksum_mismatch`

## 8. 可视化设计

为了满足“报告可视化内容丰富”，`lab/lab2/scripts/plot.py` 至少生成以下五类图：

### 8.1 运行时间折线图

- 每个版本一张
- x 轴：矩阵规模
- y 轴：平均运行时间
- 多条线：不同进程数

### 8.2 加速比图

- 基于同版本 1 进程时间计算
- x 轴：矩阵规模
- y 轴：speedup
- 多条线：不同进程数

### 8.3 并行效率图

- 计算方式：`speedup / processes`
- 用于展示扩展性衰减趋势

### 8.4 版本对比图

- 固定若干关键规模（如 `512`、`1024`、`2048`）
- 比较 `mpi_v1_p2p` 与 `mpi_v2_collective`
- 可采用分组柱状图

### 8.5 热力图

- 横轴：矩阵规模
- 纵轴：进程数
- 颜色：运行时间或 speedup
- 用于增强报告视觉表达

图像默认输出位置：

- `results/figures/*.png`
- 可选同时输出 `*.svg`

## 12. 测试与验证策略

### 12.1 小规模功能验证

对 `4x4x4`、`8x8x8` 规模支持 `--dump`，用于人工和脚本检查 A/B/C 输出。

### 12.2 跨版本一致性

在相同 `m/n/k/seed/processes` 下，`mpi_v1_p2p` 与 `mpi_v2_collective` 应满足：

- `checksum` 一致
- `max_abs` 一致或在极小浮点误差内一致

### 12.3 跨进程数一致性

同一版本在不同进程数下运行时，应与 1 进程基线的 checksum 一致。

### 12.4 自动化测试

`lab/lab2/tests/test_benchmark.py` 至少覆盖：

- benchmark 输出字段完整性
- 两个版本可运行
- benchmark 结果文件生成
- 绘图脚本可执行且图像落盘
